# Solomos Runtime Review

Πρότυπο για χειροκίνητη αξιολόγηση αποτελεσμάτων του Solomos μέσω των tests στο `tests/test_prompts.yaml`.

## 1. Πού χρησιμοποιείται

Αξιολογεί την ποιότητα γραφής που παράγει το skill σε πραγματική χρήση. Δεν αφορά τη δομική ακεραιότητα των αρχείων (αυτό το καλύπτει το `tests/static_check.py`).

Με δύο λόγια:

- `static_check.py` → ελέγχει ότι το skill είναι δομικά υγιές.
- `runtime_review.md` → ελέγχει ότι αυτό που γράφει το skill είναι ποιοτικό.

## 2. Κλίμακα βαθμολόγησης

Κάθε διάσταση παίρνει βαθμό 0-3:

- **0 = fail**: παραβίαση που χαλάει το αποτέλεσμα.
- **1 = αδύναμο**: αναγνωρίσιμο πρόβλημα, χρειάζεται αναθεώρηση.
- **2 = χρησιμοποιήσιμο**: δουλεύει με μικρές αδυναμίες.
- **3 = εξαιρετικό**: ιδανικό για το σενάριο και το register.

## 3. Διαστάσεις αξιολόγησης

Για κάθε αποτέλεσμα, βαθμολογείς:

- **Δρομολόγηση σεναρίου**: επέλεξε το σωστό σενάριο;
- **Ακρίβεια νοήματος**: διατήρησε τα facts και το νόημα;
- **Διατήρηση πρόθεσης**: κράτησε τη ρητορική δύναμη, ειρωνεία, ένταση του πρωτοτύπου;
- **Φυσικότητα ελληνικού**: διαβάζεται σαν να το έγραψε Έλληνας ομιλητής;
- **Ταίριασμα τόνου/register**: ταιριάζει με το ζητούμενο επίπεδο τυπικότητας;
- **Κρίση αγγλικών όρων**: μετέφρασε ό,τι έπρεπε και κράτησε ό,τι ήταν λεξιλόγιο του κοινού;
- **Αποφυγή υπερδιόρθωσης**: άφησε ήσυχο ό,τι ήταν ήδη φυσικό;
- **Διατήρηση σκόπιμου ύφους**: σεβάστηκε ειρωνεία, χιούμορ, τραχύτητα, φωνή χρήστη;
- **Όχι γενικός τόνος AI**: ξέφυγε από τα γνώριμα AI tics;
- **Όχι premium blandness όπου χρειάζεται δύναμη**: κρατάει ζωή σε captions, marketing, λήμματα, στοχαστική γραφή;
- **Τελική χρηστικότητα**: μπορεί ο χρήστης να το πάρει copy-paste;

## 4. Ετικέτες αστοχίας

Για κάθε αποτυχία βάζεις μία ή περισσότερες ετικέτες:

- `WRONG_SCENARIO`: λάθος σενάριο.
- `TRANSLATIONESE`: calques, αγγλική σύνταξη.
- `OVER_FORMAL`: υπερβολικά λόγιο για το register.
- `OVER_POLISHED`: υπερδιόρθωση καλού κειμένου.
- `TOO_GENERIC`: άχρωμο, θα μπορούσε να γραφτεί για οτιδήποτε.
- `PREMIUM_BLANDNESS`: καθαρό αλλά άψυχο όπου χρειάζεται δύναμη.
- `MEANING_SHIFT`: αλλαγή νοήματος ή πραγματολογικών στοιχείων.
- `INTENTION_LOSS`: έχασε τη ρητορική δύναμη του πρωτοτύπου.
- `BAD_ENGLISH_TERM_POLICY`: μετέφρασε ό,τι έπρεπε να μείνει, ή το αντίστροφο.
- `DESTROYED_STYLE`: διέλυσε ειρωνεία, χιούμορ, ή φωνή χρήστη.
- `BAD_REGISTER_MIX`: αδικαιολόγητη κατάρρευση register.
- `MISSING_INFO`: δεν ρώτησε όπου έπρεπε.
- `CLAIM_RISK`: εφεύρε αριθμούς, citations, νόμους ή άλλο που δεν υπήρχε.
- `GOOD_OUTPUT`: σχεδόν τέλειο, καμία ουσιαστική αστοχία.

## 5. Πρότυπο πίνακα αξιολόγησης

Συμπληρώνεις έναν πίνακα ανά παρτίδα:

| Test ID | Αναμενόμενο σενάριο | Σκορ | Ετικέτα αστοχίας | Σημείωση | Χρειάζεται patch; |
|---|---|---|---|---|---|
| translationese-01-condolence-hallmark | greeting-wishes |  |  |  |  |
| ... | ... |  |  |  |  |

Στο **Σκορ** βάζεις τον χαμηλότερο βαθμό μεταξύ των 11 διαστάσεων (ή τον μέσο όρο, αν προτιμάς πιο συνεχή εικόνα). Σταθερά κρατάς τη μία πολιτική σε όλη την παρτίδα.

## 6. Κανόνας απόφασης

**Δεν επιδιορθώνεις το skill βάσει ενός κακού output.** Patch μόνο όταν:

- Η αστοχία είναι σοβαρή (`CLAIM_RISK`, `MEANING_SHIFT`, παραβίαση απόλυτου κανόνα του `SKILL.md`).
- Η αστοχία επαναλαμβάνεται σε πολλά test μέσα στην ίδια κατηγορία.
- Η αστοχία προέρχεται ξεκάθαρα από συγκεκριμένο κανόνα, scenario ή exemplar που μπορείς να εντοπίσεις καθαρά.

Διαφορετικά, καταγράφεις την παρατήρηση και προχωράς. Συχνά είναι θόρυβος μιας συγκεκριμένης εκτέλεσης.

## 7. Προτεινόμενα μεγέθη παρτίδας

Ανάλογα με τον στόχο:

- **Smoke test**: 10 τυχαία tests από διαφορετικές κατηγορίες. Δείχνει αν κάτι έχει σπάσει σοβαρά.
- **Focused category run**: 10-20 tests μιας ή δύο κατηγοριών. Χρήσιμο αφού γίνει patch σε scenario, στο `judgment.md`, ή σε absolute rule.
- **Full regression pass**: όλα τα 81 tests. Πριν από major release ή κάθε φορά που αλλάζει κάποιος απόλυτος κανόνας του `SKILL.md`.

Δεν είναι αναγκαίο να γίνονται όλα συχνά. Τα tests είναι υποδομή: τα τρέχεις με σκοπό, όχι από συνήθεια.
