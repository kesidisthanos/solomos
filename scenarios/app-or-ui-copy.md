# Σενάριο: app-or-ui-copy

## Πότε χρησιμοποιείται
Κείμενα μέσα σε εφαρμογή ή ιστοσελίδα: button labels, error messages, μηνύματα συστήματος, onboarding strings, placeholder text, tooltips, push notifications, empty states, dialog titles. Διαφέρει από `marketing-copy` (δεν πουλάει), από `business-email` (δεν επικοινωνεί άνθρωπος προς άνθρωπο), από όλα τα άλλα γιατί ο χώρος είναι πολύ μικρός και η συνέπεια κρίσιμη.

## Σκοπός & κοινό
- **Σκοπός**: ο χρήστης να καταλάβει τι κάνει το κουμπί, τι πήγε στραβά, ή τι περιμένει η εφαρμογή από αυτόν, με μηδενική γνωσιακή προσπάθεια.
- **Κοινό**: ο τελικός χρήστης της εφαρμογής. Δεν διαβάζει: σαρώνει.
- **Προεπιλογή ύφους**: ουδέτερο. Φιλικό αλλά λακωνικό. Σπάνια καθημερινό (μόνο σε καταναλωτικά apps νεανικού brand).

## Δομή
Καμία. Κάθε string είναι αυτόνομο. Κρίσιμη όμως είναι η **συνέπεια** σε όλο το app: ίδια έννοια = ίδια λέξη.

## Μήκος
- Button labels: 1-3 λέξεις (ιδανικά 1-2).
- Error messages: μία πρόταση. Δύο μόνο όταν χρειάζεται οδηγία διόρθωσης.
- Push notifications: ≤80 χαρακτήρες (πολλά λειτουργικά κόβουν εκεί).
- Onboarding/empty state: 1-3 σύντομες προτάσεις.
- Tooltips: 1 πρόταση.
- Dialog titles: 2-5 λέξεις, σαφείς.

## Πρωτόκολλο πριν το γράψιμο
1. **Πού εμφανίζεται το string**; Button, dialog, banner, error, push, empty state; Καθένα έχει διαφορετικό περιορισμό μήκους και τόνου.
2. **Τι κάνει η ενέργεια**; Το label γράφεται με το ρήμα της ενέργειας («Αποθήκευση», «Αποστολή», «Διαγραφή»), όχι «OK».
3. **Ποιος ο τόνος του υπόλοιπου app**; Φιλικός, ουδέτερος, επίσημος, παιχνιδιάρης; Πρέπει να ταιριάζει.
4. **Υπάρχει αντίστοιχο string αλλού στο app**; Αν ναι, χρησιμοποίησε ίδια λέξη. Συνέπεια > δημιουργικότητα.
5. **Έχει η εταιρεία style guide**; Αν ναι, ακολούθησέ τον.
6. **Για error messages**: τι μπορεί να κάνει ο χρήστης για να το διορθώσει; Αν δεν μπορεί, μην του λες απλώς ότι έγινε λάθος, πες ότι «κάτι πήγε στραβά, δοκιμάστε σε λίγο».
7. **Πληθυντικός ή ενικός**; Διάλεξε ένα και κράτα το σε όλο το app.

## Anti-patterns ειδικά για το σενάριο
- **«Είστε σίγουρος;»** σε κάθε διαλογική ερώτηση: AI default. Φυσικό: «Διαγραφή; Δεν αναιρείται.» (ευθύ, ενημερωτικό).
- **«ΟΚ» / «Cancel» αντί ρημάτων ενέργειας**: το πρώτο button λέει τι κάνει («Αποστολή», «Διαγραφή»), το δεύτερο «Άκυρο» ή «Πίσω».
- **Calques αγγλικού UI**: «Πατήστε εδώ», «Κάντε κλικ», «Επανάληψη επιτυχής!»: γράψε «Δοκιμάστε ξανά», «Έγινε», «Αποθηκεύτηκε».
- **Τεχνική γλώσσα σε μηνύματα προς χρήστη**: «Connection timeout exceeded» → «Δεν συνδεθήκαμε. Δοκιμάστε σε λίγο.»
- **Πομπώδες φιλικό**: «Καλωσορίσατε στην εφαρμογή μας!»: μηχανικό. Πες κάτι συγκεκριμένο για το app.
- **Greeklish στα κουμπιά**: ποτέ. Είτε ελληνικά, είτε αγγλικά, ποτέ ανάμικτα στο ίδιο label.
- **Em-dashes**.
- **Πληθυντικός ευγενείας ασυνεπής**: αν ξεκίνησες με «εσείς», μην πέσεις σε «πάτα» στη μέση. Επιλέγεις πληθυντικό ή ενικό μία φορά για όλο το app.
- **Long-form copy σε μικρά UI**: αν θες να εξηγήσεις, βάλε σύνδεσμο στα docs, μη φουσκώσεις το dialog.
- **«Επιτυχής» / «Αποτυχία»** ως μόνο κείμενο: γράψε τι ακριβώς έγινε («Στάλθηκε», «Δεν στάλθηκε»).
- **Παρανόηση «error» ως «failure»**: ένα μήνυμα τύπου «Σφάλμα 503» χωρίς εξήγηση είναι άχρηστο. Πες τι σημαίνει για τον χρήστη.
- **Πολλά θαυμαστικά**: «Αποθηκεύτηκε!»: μία μονή ενημέρωση δεν χρειάζεται γιορτή. Κράτα τα θαυμαστικά για ευκαιρίες (νέο επίτευγμα, νέα χρονιά).

## Παραδείγματα

```
[Save button]
Αποθήκευση

[Confirmation dialog]
Διαγραφή του αρχείου;
Δεν αναιρείται.
[ Διαγραφή ] [ Άκυρο ]

[Empty state]
Δεν υπάρχει ακόμα τίποτα εδώ.
Πρόσθεσε την πρώτη σου εγγραφή.

[Network error]
Δεν συνδεθήκαμε στο διαδίκτυο. Δοκιμάστε σε λίγο.

[Push notification]
Νέο μήνυμα από τη Νίκη: «Είσαι ελεύθερος αύριο;»

[Toast / status]
Αποθηκεύτηκε
Στάλθηκε
Δεν στάλθηκε

[Onboarding step 1]
Ξεκινάμε.
Επίλεξε όνομα για τον λογαριασμό σου.
```

## Μορφή εξόδου
Plain text, ένα string ανά γραμμή, ή κατάλογος (αν είναι πολλά). Κάθε string με σαφή σημείωση πού πάει (button/error/etc.). Έτοιμα για paste σε i18n αρχείο ή UI editor.
